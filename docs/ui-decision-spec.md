## 🔵 시뮬레이션 ↔ 통합관리 연계 (가장 복잡한 구간)

---

**Q1. 채택된 운영항로의 스케줄 등록 시점** *(사용자 질문)*

> (1) 시나리오 생성 시 스케줄 정보 미리 입력 → 승인 시 자동 등록
> (2) 승인(채택) 완료 후 → 운영자가 통합관리에서 별도 스케줄 등록

⭕ 권장: **(2)** — 승인 완료 후 통합관리에서 등록. 이유: 시나리오는 "검토용"이고 실제 운항 일정은 확정 후 등록하는 것이 자연스러움. 단 승인 완료 시 "스케줄 등록" 바로가기 버튼 제공 필요.

---

**Q2. APPROVED_DYNAMIC_ROUTE는 ROUTE_VERSION으로 승격되는가?**

> (1) 승인된 동적경로가 별도 ROUTE_VERSION 레코드로 생성되어 기존 회랑형 항로와 동일하게 관리
> (2) APPROVED_DYNAMIC_ROUTE 테이블에만 유지되고 스케줄에서 직접 참조

→ 스케줄 등록 시 FLIGHT_SCHEDULE.RTE_SK(항로버전키)가 필요하므로 답이 결정되어야 함. 현재 DB 설계상 APPROVED_DYNAMIC_ROUTE는 별도 테이블. **스케줄이 참조할 키를 어디서 가져오는지** 결정 필요.

---

**Q3. 시나리오 생성 시 대상 항로 선택 범위**

> (1) 기존에 등록된 CORRIDOR/ROUTE_VERSION 중 선택
> (2) 신규 경로를 시나리오 내에서 직접 지도에 그려서 생성
> (3) 둘 다 가능 (탭 구분)

→ 화면 구성이 완전히 달라짐. (1)이면 드롭다운/목록, (2)이면 지도 드로잉 UI 필요.

---

**Q4. 승인 워크플로 역할 분리**

> (1) 요청자(분석 실행)와 승인자가 동일 — 본인이 분석하고 본인이 APPROVE 클릭
> (2) 요청자와 승인자가 분리 — 분석 담당자가 요청 → 관리자가 승인

→ (2)라면 승인 대기 목록 화면, 알림 기능이 별도 필요. (1)이면 단순 버튼 하나.

---

**Q5. REJECTED(반려) 시 재처리 프로세스**

> (1) 시나리오를 수정해서 재분석 실행
> (2) 완전히 새 시나리오 생성

→ (1)이면 시나리오 수정 화면 + 분석 재실행 버튼 필요. (2)이면 기존 시나리오는 이력으로만 보관.

---

**Q6. 채택(APPROVED → ACTIVE) 전환 트리거**

> (1) 승인 즉시 자동으로 ACTIVE
> (2) 스케줄 등록 완료 시점에 ACTIVE 전환
> (3) 실제 운항 시작 시각에 ACTIVE 전환 (배치/스케줄러)

→ 관제 화면에서 항로를 언제부터 실선으로 표시하는지에 직결됨.

---

## 🟢 통합관리 서비스 내부

---

**Q7. 스케줄 변경 시 이전 버전 처리**

> (1) 기존 스케줄 레코드를 수정 (덮어쓰기)
> (2) 새 버전 레코드 생성 + 기존 버전 비활성화 (SCHDL_VRSN_NO 증가)

→ DB에는 버전 컬럼(SCHDL_VRSN_NO)이 있으나, 화면에서 이력 조회를 제공할지 결정 필요.

---

**Q8. 반복 규칙(RECR_RULE) 입력 UI 형태**

> (1) 텍스트로 직접 입력 (예: `MON,WED,FRI 09:00`)
> (2) 요일 체크박스 + 시간 선택 위젯으로 조합 → 내부적으로 룰 문자열 생성

→ 비개발자 운영자가 사용하므로 (2)가 UX상 적절하나 구현 복잡도 증가.

---

**Q9. 장애물(OBSTACLE) 등록 우선순위**

> (1) API 자동수집 데이터가 항상 우선 (수동 등록 불가)
> (2) 수동 등록 가능, API와 충돌 시 수동 우선
> (3) 수동 등록 가능, API와 충돌 시 최신 업데이트 우선

→ 화면에 수동 등록 폼을 제공할지, 원천유형(SRC_TYPE_CD) 배지를 표시할지 결정.

---

**Q10. EAV 속성 항목 자체(AIRCRAFT_ATTRIBUTE / VERTIPORT_ATTRIBUTE) 관리 화면**

> (1) 별도 "속성 항목 관리" 메뉴 존재 (속성명·데이터유형 추가/삭제)
> (2) 개발자가 직접 DB에 삽입 (화면 미제공)

→ (1)이면 DT-MENU에 항목 추가 필요. 현재 메뉴 목록에는 미포함.

---

## 🔴 관제 서비스 내부

---

**Q11. TRACK_EVENT_LOG 생성 주체**

> (1) 시스템이 자동 감지해서만 생성 (이탈·이상 감지 알고리즘)
> (2) 관제사가 수동으로 이벤트 등록 가능

→ (2)라면 관제 화면에 수동 이벤트 등록 폼 필요.

---

**Q12. 이상상황 알림 확인/처리 흐름**

> (1) 알림 확인만 하면 종료 (조치상태 자동 RESOLVED)
> (2) 알림 확인 → 조치 내용 입력 → RESOLVED 처리 (ACTN_STTS_CD 변경)

→ (2)라면 이벤트 상세 팝업에 조치 입력 필드 필요.

---

**Q13. 관제 화면의 실시간 갱신 방식**

> (1) 폴링 (일정 주기로 API 재호출)
> (2) WebSocket/SSE (서버 푸시)

→ 화면 기획 상 명시적 차이는 없으나, "자동 갱신 주기 표시기" UI 필요 여부에 영향.

## 🔁 서비스 간 화면 전환 (UX 플로우)

---

**Q14. 시뮬레이션 결과 → 통합관리 이동 방식**

> (1) 채택 버튼 클릭 시 통합관리 > 항로 일정관리로 자동 이동
> (2) 채택은 시뮬레이션에서 완결, 별도 메뉴 이동 없음
> (3) 채택 후 "스케줄 등록하기" 바로가기 배너/버튼 제공 (이동 여부는 선택)

---

**Q15. 관제 화면에서 스케줄 미등록 항로 표시 방식**

> (1) 미표시 (스케줄 있는 항로만 관제 화면에 표출)
> (2) 점선 또는 비활성 색상으로 표시
> (3) 레이어 토글로 ON/OFF

---

**Q16. 대시보드 데이터 기준 시점**

> (1) 실시간 (LATEST_POSITION, VERTIPORT_OP_STATUS 기준)
> (2) 일 단위 집계 (MART_ROUTE_HAZARD 기준)
> (3) 혼합 (KPI는 실시간, 통계는 집계)

→ 새로고침 버튼 필요 여부, "기준 시각" 표시 위치에 영향.
