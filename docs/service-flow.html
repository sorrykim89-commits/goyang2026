1. 전체 서비스 흐름

UAM 시스템은 크게 3개 서비스 영역과 3개 데이터 흐름으로 구성된다.

데이터 흐름 요약

흐름 방향 설명
수집 흐름 외부 → DB 기상/NOTAM/공역 등 외부 API + 관제/텔레메트리 스트림
관리 흐름 사용자 ↔ DB 기체/버티포트/스케줄 등 운영자가 직접 CRUD
분석 흐름 DB → AI → DB 마스터+환경 데이터로 분석 → 결과를 DB에 저장 → 승인 후 운영 반영

2. 관제 서비스 흐름 (Real-time Layer)

실시간 상황 인지 — "지금 기체가 어디에 있고, 위험요소는 무엇인가"

관제 서비스 데이터 흐름 

기체 텔레메트리 ──Stream──→ POSITION(시계열) ──집계──→ LATEST_POSITION(캐시)
                                                          ↓
관제시스템 ─────Stream──→ FLIGHT_ACTUAL(운항상태)    관제화면: 기체위치 표출
                               ↓
                    이상감지 → TRACK_EVENT_LOG → 알림 표출

기상/NOTAM/재난 ──API──→ WEATHER/NOTAM/DISASTER → MART_ROUTE_HAZARD → 위험 레이어 표출

3. 시뮬레이션 서비스 흐름 (Planning Layer)

사전 검토·분석 — "이 항로가 안전한가, 이 버티포트가 적합한가"

시뮬레이션 핵심 워크플로 

[시나리오 생성] → [AI 분석] → [결과 저장] → [승인 워크플로] → [운영 항로 승격]

상세:
1. 사용자가 조건 설정 (항로, 기상, 기체, 룰셋)
2. SIMULATION_SCENARIO 생성
3. AI 엔진이 분석 수행:
   - 경로 후보 생성 (ROUTE_CANDIDATE → DYNAMIC/ALTERNATE)
   - 각 후보 평가 (ROUTE_EVAL_RESULT)
   - 안전성 분석 (ANALYSIS_RESULT)
   - 대체 버티포트 추천 (ALT_VERTIPORT_REC)
4. 자동 보고서 생성 (REPORT_META)
5. 승인 워크플로: PENDING → APPROVED → ACTIVE
6. 승인된 경로 → APPROVED_DYNAMIC_ROUTE / APPROVED_ALTERNATE_ROUTE

4. 통합관리 서비스 흐름 (Operation Layer)

운영·스케줄 관리 — "기체/버티포트/스케줄을 어떻게 관리하는가"

EAV 동적 속성 관리 흐름

5. 데이터 원천별 흐름 종합

6. 서비스 간 연계 관계

연계 방향 내용
통합관리 → 관제 마스터 제공 기체/항로/버티포트 기준정보가 관제 화면의 기반
통합관리 → 시뮬레이션 입력 제공 마스터 + 스케줄이 시뮬레이션의 입력 조건
시뮬레이션 → 통합관리 결과 반영 승인된 항로가 운영 항로로 승격
시뮬레이션 → 관제 기준 제공 안전 룰셋/위험 기준이 이상 감지 기준
관제 → 통합관리 실적 축적 운항 실적/이벤트가 통계·대시보드에 반영
